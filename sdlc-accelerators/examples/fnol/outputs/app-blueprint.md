## §1. Application Overview

**Solution:** FNOL Claims Agent — First Notice of Loss claims intake automation for auto insurance.

**Use Case:** When a customer calls to report an auto accident, the FNOL agent automates the intake process: extracting claim details from unstructured input (phone/web), enriching the claim with policy, vehicle, and weather data from 3 sources in parallel, classifying severity using business rules, delegating body shop estimates to a partner's A2A agent, and routing high-severity claims to a human adjuster for review.

```mermaid
    Coord->>Par: delegate(extracted_details)
    par Parallel enrichment
        Par->>Policy: enrich(policy_no)
        Policy-->>Par: policy_details
    and
        Par->>Vehicle: enrich(vin)
        Vehicle-->>Par: vehicle_details
    and
        Par->>Weather: enrich(location, date)
        Weather-->>Par: weather_conditions
    end
    Par-->>Coord: enriched_claim

    Note over Coord: Step 3: Classify severity
    Coord->>Sev: classify(enriched_claim)
    Sev-->>Coord: severity=HIGH

    Note over Coord: Step 4: Human review (HIGH severity)
    Coord->>HR: route_to_human(enriched_claim, severity=HIGH)
    HR-->>Coord: review_decision{approved, adjuster_notes}

    Coord-->>GW: claim_response{claim_id, status, severity, review}
    GW-->>User: 200 OK + claim summary
```

## §3. Architecture Patterns Used

| Pattern | Role | Nesting | Confidence | Source Signal |
|---|---|---|---|---|
| SequentialAgent | Root orchestrator | — | 0.95 | "First the agent extracts... then it enriches... finally it classifies" |
| ParallelAgent | Parallel enrichment (3 sources) | Inside SequentialAgent (step 2) | 0.90 | "In parallel, it enriches from three sources" |
| LoopAgent | Retry on enrichment failure | Inside ParallelAgent (per source) | 0.88 | "If any enrichment fails, retry up to 3 times" |
| CustomAgent (HITL) | Human review for high-severity | Inside SequentialAgent (step 4) | 0.85 | "A human reviews high-severity claims" |

**Adjacency validation:** ✅ Passed. LoopAgent inside ParallelAgent is valid per composition rules. LoopAgent does NOT nest inside another LoopAgent.

## §6. HA/DR Guidance

**DR Strategy:** Pilot Light — Cold Standby (selected in plan.md, Tier-2 application)

### GCP Product-Level HA/DR Guidance

| Component | HA (within region) | DR (cross-region) | RPO | RTO |
|---|---|---|---|---|
| Cloud Run | Auto-restart, multi-zone | Cold standby in DR region, scale-up on failover | 0 (stateless) | 5-10 min (container pull + scale) |
| Cloud SQL | Automatic failover to zonal standby | Cross-region read replica, promote on failover | 1 hour (async replication) | 15-30 min (promote + DNS) |
| Apigee Gateway | Multi-zone load balancing | Multi-region deployment, DNS failover | 0 (stateless) | 2-5 min (DNS TTL) |
| Secret Manager | Regional replication | Global replication (automatic) | 0 (global) | 0 (global) |
| MCP Servers | Health check + restart | DR region endpoints pre-configured | varies | varies |

**Aggregate RTO:** 30-60 minutes (dominated by Cloud SQL promote + Cloud Run scale-up)
**Aggregate RPO:** 1 hour (dominated by Cloud SQL async replication lag)

### Failover Decision Matrix

| Scenario | Trigger | Action | Estimated Duration |
|---|---|---|---|
| Single container crash | Cloud Run health check | Auto-restart in same zone | < 30 seconds |
| Zone failure | Regional health check | Cloud Run redistributes to healthy zones | 1-2 minutes |
| Region failure | Manual or automated (Cloud Monitoring alert) | Promote DR read replica, scale up DR Cloud Run, DNS failover | 30-60 minutes |
| Region recovery | Manual verification | Reverse replication, scale down DR, restore primary | 1-2 hours |
