# Eraser.io DSL Input — FNOL Claims Agent

## How the DSL is Constructed

The Blueprint Advisor's background pipeline constructs the Eraser.io DSL from three sources, in this order:

```
Source 1: spec.md
  → §2 Workflow ordering words ("first... then... in parallel... finally")
  → §4 Data Sources (5 MCP servers)
  → §5 External Partners ("they operate their own system" → A2A)
  → §7 Business Rules (FunctionTools)

Source 2: plan.md
  → Runtime (Cloud Run)
  → Gateway (Apigee)
  → Security (VPC-SC, CMEK, Model Armor)
  → DR Strategy (Pilot Light — Cold Standby, us-east1 / us-west1)

Source 3: Blueprint Advisor reasoning outputs (during background pipeline)
  → MS1 pattern-composition: Sequential(Parallel(Loop×3), HITL) — confidence 0.95
  → MS2 data-platform-selection: 5 MCP servers discovered from Tool Catalog
  → MS3 agent-boundary: body-shop-a2a discovered from API Hub (A2A boundary)
  → MS4 skill-tool-discovery: severity_classifier_fn, coverage_calculator_fn
  → Agent Identity derivation: per-agent capabilities/denied/delegation
```

The pipeline assembles the DSL BEFORE `assemble_blueprint` runs — the diagrams are generated first, then bundled into the blueprint result alongside the .md and .json.

## What Eraser.io Returns

For each DSL input, the Eraser.io API returns:

```
Input:  Eraser.io DSL (text above)
   ↓
Output 1: .eraser file (Eraser.io native format — DISCARDED, not delivered to workspace)
Output 2: .png (rendered preview — delivered, referenced in app-blueprint.md as ![](diagrams/*.png))
Output 3: .drawio.xml (editable — delivered, developer opens in Draw.io VSCode extension)
```

The pipeline calls Eraser.io twice per blueprint:
1. Component topology DSL → `component-topology.drawio.xml` + `.png`
2. HA/DR lifecycle DSL → `hadr-lifecycle.drawio.xml` + `.png`

## Where in the Pipeline This Happens

```
blueprint_start(spec, plan)
    ↓
Background pipeline:
    Step 0: validate_spec (quality gate)
    Step 1: Pattern composition (MS1) → Sequential + Parallel + Loop + HITL
    Step 2: Data platform selection (MS2) → 5 MCP servers
    Step 3: Agent boundary (MS3) → body-shop-a2a (A2A)
    Step 4: Skill/tool discovery (MS4) → 2 FunctionTools
    Step 5: recommend_architecture (RAG)
    Step 6: adr_compliance_check
    Step 7: resolve_modules (GitHub)
    ★ Step 8: CONSTRUCT Eraser.io DSL from Steps 1-7 outputs    ← THIS IS WHERE THE DSL IS BUILT
    ★ Step 9: CALL Eraser.io API with component topology DSL    ← FIRST CALL
    ★ Step 10: CALL Eraser.io API with HA/DR lifecycle DSL      ← SECOND CALL
    Step 11: assemble_blueprint (.md + .json from all above)    ← DIAGRAMS ALREADY EXIST
    Step 12: bundle .md + .json + .drawio.xml + .png → Task Store
    ↓
blueprint_result(taskId) → delivers everything to workspace
```

The DSL construction (Step 8) is a **deterministic assembly** — no LLM involved. It takes the structured outputs from Steps 1-7 (agent names, types, parent-child relationships, tool bindings, A2A endpoints, infrastructure modules, DR strategy) and formats them as Eraser.io DSL text. The LLM reasoning happened in Steps 1-6; the DSL is just a serialization format for the reasoning results.
