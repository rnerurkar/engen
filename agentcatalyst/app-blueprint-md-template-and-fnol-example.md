# app-blueprint.md — Template and FNOL Reference Example

## 1. Template Structure

The Blueprint Advisor produces `app-blueprint.md` — a 9-section governance document that serves as:
- The **single source of truth** for architectural decisions reviewed by the SA and EA
- The **input to Governance Guardian** (`/catalyst.assess` extracts all artifacts directly from this file)
- The **source for `app-blueprint.json`** (machine-readable spec generated by `assemble_blueprint` — the coding agent reads `.json`, never `.md`)
- A **human-readable review artifact** that renders natively in GitHub and VSCode

### Why 9 sections (not 18)

The previous 18-section format split into Part I (§1-§7 governance) and Part II (§8-§18 technical). Part II was auto-generated from Part I and silently overwritten on refresh — creating a trap where edits to Part II sections appeared to work but were lost. The 9-section format eliminates this:

- **Every section is genuinely editable.** No section is auto-regenerated from another section.
- **Technical config lives in `.json` only.** Agent topology, tool bindings, MCP configs, infrastructure modules, business rules, eval config, screening config, and pipeline configs are derived by `assemble_blueprint` from `spec.md` + `plan.md` + the Blueprint Advisor's reasoning and written directly into `app-blueprint.json`. The developer never sees pseudo-editable technical sections that get overwritten.
- **The SA reviews what matters.** Application overview, architecture patterns, tech stack, DevSecOps stack, HA/DR, decision log, and NFRs — the governance decisions that EA assesses.

### Workspace file layout

When `blueprint_result` delivers the blueprint, the coding agent writes the following files:

```
features/fnol-claims-agent/
├── app-blueprint.md                    ← PRIMARY: 9-section governance document
├── diagrams/
│   ├── component-topology.drawio.xml   ← Editable in Draw.io VSCode extension (converted from .eraser by Eraser.io)
│   ├── component-topology.png          ← Rendered preview (inline in §2)
│   ├── hadr-lifecycle.drawio.xml       ← Editable in Draw.io VSCode extension
│   └── hadr-lifecycle.png              ← Rendered preview (inline in §7)
└── ... (spec.md, plan.md already here)
```

**No `app-blueprint.json` at this stage.** The `.json` is generated only when `/catalyst.assess` or `/catalyst.generate` triggers `assemble_blueprint` (which derives the machine-readable spec from the `.md` + `spec.md` + `plan.md`).

### Diagram generation pipeline (Eraser.io headless)

The Blueprint Advisor's background pipeline generates diagrams using the **Eraser.io headless API**:

1. **Eraser.io DSL generation:** The pipeline assembles an Eraser.io DSL description from the agent topology, MCP connections, A2A boundaries, and infrastructure components inferred from `spec.md` + `plan.md`.
2. **Rendering:** Eraser.io API renders the DSL to `.eraser` format + `.png` preview.
3. **Conversion:** Eraser.io API converts `.eraser` → `.drawio.xml` (editable in Draw.io VSCode extension).
4. **Delivery:** Only `.drawio.xml` + `.png` are delivered to the workspace. The `.eraser` intermediate is not retained.

The same pipeline runs for HA/DR lifecycle diagrams (one diagram per DR strategy with 4 lifecycle scenarios).

**Mermaid fallback:** If the Eraser.io API is unreachable, the Blueprint Advisor generates inline mermaid diagrams in §2 (component) and §7 (HA/DR). Mermaid renders natively in VSCode and GitHub. The `.drawio.xml` and `.png` files are absent but the architecture is captured in text. When Eraser.io recovers, `/catalyst.refresh` regenerates the diagram files.

The following content is NOT in `app-blueprint.md` — it is derived by `assemble_blueprint` from `spec.md` + `plan.md` + the Blueprint Advisor's reasoning and written directly into `app-blueprint.json`:

### Sequence Diagram — Happy Path

```mermaid
sequenceDiagram
    autonumber
    participant User as Adjuster
    participant GW as Apigee Gateway
    participant Coord as fnol_coordinator
    participant Extract as extract_details
    participant DB as claims-db-mcp
    participant Par as parallel_enrichment
    participant Policy as enrich_policy
    participant Vehicle as enrich_vehicle
    participant Weather as enrich_weather
    participant Sev as severity_classifier
    participant HR as human_review

    User->>GW: POST /api/v1/claims (unstructured claim text)
    GW->>Coord: Route to fnol_coordinator

    Note over Coord: Step 1: Extract details
    Coord->>Extract: delegate(claim_text)
    Extract->>DB: query existing policy
    DB-->>Extract: policy_record
    Extract-->>Coord: extracted_details{policy_no, vin, location, date, damage_est}

    Note over Coord: Step 2: Parallel enrichment
    Coord->>Par: delegate(extracted_details)
    par Parallel enrichment
        Par->>Policy: enrich(policy_no)
        Policy-->>Par: policy_details
    and
        Par->>Vehicle: enrich(vin)
        Vehicle-->>Par: vehicle_details
    and
        Par->>Weather: enrich(location, date)
        Weather-->>Par: weather_conditions
    end
    Par-->>Coord: enriched_claim

    Note over Coord: Step 3: Classify severity
    Coord->>Sev: classify(enriched_claim)
    Sev-->>Coord: severity=HIGH

    Note over Coord: Step 4: Human review (HIGH severity)
    Coord->>HR: route_to_human(enriched_claim, severity=HIGH)
    HR-->>Coord: review_decision{approved, adjuster_notes}

    Coord-->>GW: claim_response{claim_id, status, severity, review}
    GW-->>User: 200 OK + claim summary
```

## §3. Architecture Patterns Used

| Pattern | Role | Nesting | Confidence | Source Signal |
|---|---|---|---|---|
| SequentialAgent | Root orchestrator | — | 0.95 | "First the agent extracts... then it enriches... finally it classifies" |
| ParallelAgent | Parallel enrichment (3 sources) | Inside SequentialAgent (step 2) | 0.90 | "In parallel, it enriches from three sources" |
| LoopAgent | Retry on enrichment failure | Inside ParallelAgent (per source) | 0.88 | "If any enrichment fails, retry up to 3 times" |
| CustomAgent (HITL) | Human review for high-severity | Inside SequentialAgent (step 4) | 0.85 | "A human reviews high-severity claims" |

**Adjacency validation:** ✅ Passed. LoopAgent inside ParallelAgent is valid per composition rules. LoopAgent does NOT nest inside another LoopAgent.

**Pattern source:** Company Pattern Catalog (Vertex AI Search) — 11 agentic patterns, matched by spec ordering signals.

## §6. HA/DR Guidance

**DR Strategy:** Pilot Light — Cold Standby (selected in plan.md, Tier-2 application)

### GCP Product-Level HA/DR Guidance

| Component | HA (within region) | DR (cross-region) | RPO | RTO |
|---|---|---|---|---|
| Cloud Run | Auto-restart, multi-zone | Cold standby in DR region, scale-up on failover | 0 (stateless) | 5-10 min (container pull + scale) |
| Cloud SQL | Automatic failover to zonal standby | Cross-region read replica, promote on failover | 1 hour (async replication) | 15-30 min (promote + DNS) |
| Apigee Gateway | Multi-zone load balancing | Multi-region deployment, DNS failover | 0 (stateless) | 2-5 min (DNS TTL) |
| Secret Manager | Regional replication | Global replication (automatic) | 0 (global) | 0 (global) |
| MCP Servers | Health check + restart | DR region endpoints pre-configured | varies | varies |

**Aggregate RTO:** 30-60 minutes (dominated by Cloud SQL promote + Cloud Run scale-up)
**Aggregate RPO:** 1 hour (dominated by Cloud SQL async replication lag)

### Failover Decision Matrix

| Scenario | Trigger | Action | Estimated Duration |
|---|---|---|---|
| Single container crash | Cloud Run health check | Auto-restart in same zone | < 30 seconds |
| Zone failure | Regional health check | Cloud Run redistributes to healthy zones | 1-2 minutes |
| Region failure | Manual or automated (Cloud Monitoring alert) | Promote DR read replica, scale up DR Cloud Run, DNS failover | 30-60 minutes |
| Region recovery | Manual verification | Reverse replication, scale down DR, restore primary | 1-2 hours |

### How `assemble_blueprint` generates `app-blueprint.json`

`assemble_blueprint` reads `app-blueprint.md` (for §1-§9 governance content) PLUS `spec.md` (for agent topology, tool bindings, business rules) and `plan.md` (for infrastructure, CI/CD, eval config). It produces `app-blueprint.json` with all fields the coding agent needs — including the technical config that is NOT in the `.md`:

```
app-blueprint.md (§1-§9)  ──┐
spec.md (topology, tools,    ├──→ assemble_blueprint ──→ app-blueprint.json
  business rules, NFRs)     │        (deterministic,     (adk_agent_tree,
plan.md (infra, CI/CD,      │         no LLM)             tool_bindings,
  DR, eval, security)  ─────┘                             infra_modules,
                                                           business_rules,
                                                           pipeline_configs,
                                                           screening_config,
                                                           eval_config,
                                                           confidence_scores)
```

The `.json` is regenerated every time `/catalyst.refresh`, `/catalyst.assess`, or `/catalyst.generate` detects that the `.md` has changed. The developer never edits `.json` directly.
